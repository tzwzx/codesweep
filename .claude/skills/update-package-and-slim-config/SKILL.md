# codesweep プロジェクトのリフレッシュ

codesweep プロジェクトの npm 依存関係を最新バージョンに更新し、リリースノートを確認し、設定ファイルの冗長な記述を削除して、プロジェクト全体をクリーンな状態にする。

このスキルは **codesweep リポジトリ専用**。codesweep は Expo でも Next.js でもない汎用 TypeScript CLI なので、フレームワーク判定は不要。`.github` ディレクトリも Git URL 依存も持たないため、GitHub Actions 更新・Git URL パッケージ更新のフェーズは対象外。元になった汎用スキルからこの2フェーズを削ぎ落とし、codesweep の構成（bun / ultracite ベースの設定 / 自己ホスティングな検証）に合わせて固定したのがこのスキル。

## フェーズ全体像

1. **npm パッケージ更新** — 全依存を最新バージョンへ
2. **リリースノート確認・不具合リスク評価** — 更新内容を調査してレポート
3. **ベストプラクティス移行** — 新バージョンが推奨する書き方へ移行
4. **設定ファイルのスリム化** — 冗長な設定を検出・削除
5. **検証** — ビルドと codesweep でパスを確認

各フェーズは上から順に実行する。Phase 2〜4 で生成した個別レポートは、最後に1つの「リフレッシュレポート」としてまとめてユーザーに提示する。

---

## Phase 1: npm パッケージ更新

codesweep の依存はすべて exact バージョン（`.syncpackrc.json` の semverGroup がレンジ指定を禁止している）。`--removeRange` を付けることでこの方針を維持したまま更新する。

```bash
bunx npm-check-updates -u --removeRange && rm -rf node_modules bun.lock && bun i
```

更新後、変更内容を確認する：

```bash
git diff package.json
```

この差分から更新されたパッケージとバージョンの変化を抽出し、Phase 2 に渡す。**更新可能なパッケージが1つもなかった場合**は、その旨を記録して Phase 4 へ進む（Phase 2・3 はスキップ）。

---

## Phase 2: リリースノート確認・不具合リスク評価・レポート

`git diff package.json` でバージョンが変わったパッケージを特定し、**すべての更新パッケージ**についてリリースノートを確認する。

### 確認方法

1. **Context7 MCP** を使って、更新されたライブラリの最新ドキュメントを確認する
2. Context7 で十分な情報が得られない場合は **WebSearch** で `{パッケージ名} changelog` や `{パッケージ名} release notes {新バージョン}` を検索する
3. それでも見つからない場合は GitHub の Releases ページを WebFetch で確認する

旧バージョンと新バージョンの**間に入る全リリース**の公式 changelog エントリを確認する（複数バージョンを跨ぐ場合は中間バージョンも含む）。記憶や推測で「だいたいこういう変更」と書かず、確認できた公式の記載のみに基づいて報告する。同じバージョンを別々の言葉で曖昧に要約してしまう事故を防ぐため、確認に使った出典（changelog の該当バージョン項目、GitHub Releases の URL 等）が後から追える形にする。出典が見つからなかったパッケージは、レポートにその旨を明記する。

### レポート対象の判断基準

すべての変更をレポートする必要はない。利用者に影響がある変更のみを報告する：

**レポートする：**

- 破壊的変更（API 変更、削除、動作変更）
- 新しい公開 API・オプションの追加
- 非推奨（deprecation）の通知
- 設定ファイルやビルド設定の変更が必要なもの
- セキュリティ修正

**レポートしない（スキップ）：**

- 内部的なリファクタリングや最適化
- テストの追加・修正
- ドキュメントのみの変更
- 内部依存関係の更新
- ユーザーのコードに影響しないバグ修正

パッチ更新で利用者に影響がある変更がない場合は、そのパッケージの詳細説明は不要。一覧表にバージョン番号だけ記載すればよい。

### 不具合リスク評価（dependencies のみ）

codesweep の `dependencies` は **`concurrently` と `yaml` の2つだけ**。これらは codesweep の**ランタイム依存**（`runner.ts` の並列実行に `concurrently`、`config.ts` の YAML パースに `yaml`）であり、破壊的変更は codesweep を使う側に直接影響する。一方、devDependencies（typescript / oxlint / oxfmt / knip / syncpack / ultracite / @types/node）はビルド・開発時のみ使うツールなので、不具合リスク評価の対象外。

`concurrently` または `yaml` が更新された場合は、以下の観点で調査し、レポート末尾に評価を記載する：

- **内部動作の変更**: パッチ更新でも内部的なデフォルト値変更・タイミング変更がないか
- **既知の不具合**: 新バージョンで報告されている既知の issue やリグレッションがないか（GitHub Issues を確認）
- **実行環境への影響**: Node.js >= 18 / Bun サポートに影響する変更がないか

各パッケージに以下のリスクレベルを付与する：

- 🟢 **低リスク**: ビルド修正・内部リファクタのみ。公開 API やランタイム動作に変更なし
- 🟡 **中リスク**: 動作変更を含むが、codesweep の利用箇所には影響しない可能性が高い
- 🔴 **高リスク**: codesweep が利用している API や動作に直接影響する変更がある

### レポート形式

```
## 📦 パッケージ更新レポート

### ⚠️ 破壊的変更あり
| パッケージ | 旧バージョン | 新バージョン | 内容 |
|-----------|------------|------------|------|
| yaml      | 2.9.0      | 3.0.0      | parse の○○が変更... |

### ✨ 新機能・改善
| パッケージ | 旧バージョン | 新バージョン | 内容 |
|-----------|------------|------------|------|
| typescript | 6.0.3     | 6.1.0      | ○○オプションが追加... |

### 📋 その他の更新（利用者への影響なし）
| パッケージ | 旧バージョン | 新バージョン |
|-----------|------------|------------|
| oxlint    | 1.64.0     | 1.65.0     |

### ❌ 更新できなかったパッケージ
| パッケージ | 理由 |
|-----------|------|
| xxx       | ○○との互換性問題 |

### 🛡️ dependencies 不具合リスク評価
| パッケージ | バージョン変更 | リスク | 評価 |
|-----------|-------------|-------|------|
| concurrently | 9.2.1 → 9.2.2 | 🟢 低 | バグ修正のみ。公開 API 変更なし |

**根拠:**
- **concurrently 9.2.2**: ○○の理由により安全と判断...
```

破壊的変更がある場合は、必要に応じて `src/` 内のコードも修正する。Context7 MCP で最新の実装方法を確認し、新しい書き方に合わせる。

## Phase 3: ベストプラクティス移行

リリースノートで得た知識をもとに、`src/` のコードを新バージョンが推奨する書き方に合わせる改善を提案・適用する。バージョンアップは番号の更新だけでなく、新バージョンが提供する最適な書き方へ移行する機会でもある。

### 対象となる改善

- **新 API への移行**: 旧 API が非推奨になった、またはより良い新 API が追加された場合
- **設定の最適化**: 新バージョンで推奨される設定値・オプションへの変更（特に `tsconfig.json` のコンパイラオプション）
- **コードパターンの改善**: より簡潔で推奨される書き方への移行
- **型安全性の向上**: 新しい型定義やユーティリティ型の活用

### 進め方

1. Context7 MCP で各パッケージの最新ドキュメント・マイグレーションガイドを確認する
2. `src/` 内で旧パターンが使われている箇所を Grep で検索する
3. 改善が見つかった場合、以下の形式でレポートに追加する：

```
### 🚀 ベストプラクティス移行

#### typescript (6.0.3 → 6.1.0)
- **新コンパイラオプション `xxx`**
  該当箇所: `tsconfig.json`
  → ○○することで型チェックが厳密になります
```

4. 影響範囲が小さく明確に改善となるものは自動で適用する。判断に迷うもの（大規模なリファクタが必要、トレードオフがある等）はレポートに記載して提案にとどめる。

## Phase 4: 設定ファイルのスリム化

パッケージ更新後は、デフォルト値や extends 元の内容が変わっている可能性がある。このタイミングで設定ファイルの冗長な記述を検出・削除する。

### codesweep の設定ファイル

ルートにある設定ファイルは以下。それぞれツールと extends 元が決まっているので、汎用スキルのようなファイル発見プロセスは不要。

| ファイル           | ツール     | extends 元              | デフォルト値の確認先                                                          |
| ------------------ | ---------- | ----------------------- | ----------------------------------------------------------------------------- |
| `oxfmt.config.ts`  | oxfmt      | `ultracite/oxfmt`       | `node_modules/ultracite/`（oxfmt エクスポート）+ `node_modules/oxfmt/`        |
| `oxlint.config.ts` | oxlint     | `ultracite/oxlint/core` | `node_modules/ultracite/`（oxlint core エクスポート）+ `node_modules/oxlint/` |
| `knip.json`        | knip       | なし                    | `node_modules/knip/` のスキーマの `default`                                   |
| `.syncpackrc.json` | syncpack   | なし                    | `node_modules/syncpack/schema.json` の `default`                              |
| `tsconfig.json`    | TypeScript | なし                    | TypeScript コンパイラのデフォルト                                             |

`codesweep.yml` は codesweep 自身のチェック定義であり、プロジェクト固有の意図的な設定。スリム化の対象外。

### 重要：ultracite 更新後の再点検

`oxfmt.config.ts` と `oxlint.config.ts` は `ultracite` から設定を継承している。Phase 1 で `ultracite` が更新されると extends 元の内容が変わるため、**codesweep がローカルで再定義している設定が extends 元と重複していないか**を必ず再点検する。

例：`ultracite/oxfmt` が `dist/**` をデフォルトで無視するようになれば、`oxfmt.config.ts` の `ignorePatterns` から `dist/**` は冗長になる。

### 冗長な設定の定義

以下の3パターンを「冗長」として削除対象にする：

1. **デフォルト値と同じ設定** — ツールが明示的な設定なしでも同じ動作をするもの
2. **extends 元と同じ値での上書き** — 継承した設定を同じ値で再定義しているもの
3. **不要なルール無効化・無視指定** — `"off"` 設定や ignore パターンで、現在の状態では不要になっているもの

逆に、以下は冗長に見えても削除しない：

- **デフォルト値だが意図的に明示している設定** — ただし extends 元と同じ値での上書きはこの例外に当てはまらない
- **確認が取れない設定** — デフォルト値やマージ戦略が不明なもの
- **意図的なルール無効化** — プロジェクト方針として無効にしているルール
- **ビルド出力・生成物の ignore** — `dist/**` のような成果物の無視指定。`dist/` はリポジトリにコミットされる成果物だが、lint・フォーマットの対象にはしたくないため ignore は必要

### 4a. 分析

設定ファイルは少数なのでインラインで分析してよい。`node_modules` 内のデフォルト値・extends 元の調査で深掘りが必要な場合のみ Explore サブエージェントを使う。

各ファイルについて、ツールの特定 → デフォルト値の確認 → extends 元の確認を行う。

**デフォルト値の確認** — 以下のソースを優先順位順にチェック：

1. **JSON Schema** — `node_modules/<tool>/` のスキーマファイルの `default` フィールド
2. **ソースコード** — `node_modules/<tool>/` 内のデフォルト定義
3. **型定義** — JSDoc コメントや `@default` アノテーション

スキーマやソースコードから確認できないものは、削除対象から除外する。推測で削除しない。

**extends 元の確認** — `oxfmt.config.ts` / `oxlint.config.ts` は `ultracite` の継承元を `node_modules/ultracite/` から読み、継承元の値と現在のファイルの値を比較する。同じ値で上書きしている箇所を特定する。

**マージ戦略の確認** — ツールによって配列・オブジェクトのマージ戦略が異なる：

| 戦略                | 動作                     | 子で省略した場合         |
| ------------------- | ------------------------ | ------------------------ |
| **置換（replace）** | 子の値が親を完全に上書き | 親の値がそのまま使われる |
| **マージ（merge）** | 親と子の値が結合される   | 親の値のみが使われる     |

マージ戦略が不明な場合はスキーマやドキュメントで確認する。確認できなければ削除しない。

**ignore 指定の検証** — `ignorePatterns` / `ignore` の各パターンについて、(1) 対象ファイル・ディレクトリが実在するか、(2) extends 元で既に無視されていないか、を確認する。何もマッチしないパターンや extends 元でカバー済みのパターンは削除候補。ただしビルド出力（`dist/**`）の ignore は上記のとおり残す。

**codesweep の設定ファイルは元々ミニマル**に保たれているため、削除すべき冗長設定が見つからないことも多い。その場合は無理に削除点を作らず「変更なし」と正直に報告する。

### 4b. 結果の提示

```
## 🧹 設定ファイル スリム化レポート

### 削除対象
| ファイル | 削除対象 | 理由 |
|----------|---------|------|
| `oxlint.config.ts` | `ignorePatterns: ["dist/**"]` | ultracite/oxlint/core が dist/** をデフォルトで無視するようになったため重複 |

### 変更なし（すべて必要な設定）
- `oxfmt.config.ts` — `ignorePatterns` の `dist/**`・`.vscode/**` はビルド出力・エディタ設定の保護に必要
- `.syncpackrc.json` — exact バージョン強制は codesweep のプロジェクト固有方針
- `knip.json` — `ignore: ["dist/**"]` はビルド出力を未使用検出から除外するため必要
- `tsconfig.json` — すべてのコンパイラオプションが意図的な設定
```

「理由」列は具体的に書く。「デフォルトだから」ではなく「ultracite/oxlint/core が dist/\*\* をデフォルトで無視するようになったため」のように、確認した根拠を明示する。

### 4c. 編集

Edit ツールで冗長な設定を削除する。

- `.config.ts` / JSON / JSONC のフォーマットが崩れないよう注意する
- 設定に付随するコメントも一緒に削除する
- 配列の要素を1つ削除して空配列になった場合、そのプロパティ自体を削除してよいか（デフォルトが空配列か）を確認する

## Phase 5: 検証

codesweep は自分自身の CLI（`dist/cli.js`）でチェックを実行する自己ホスティング構成。`codesweep:check` / `codesweep:fix` は `node dist/cli.js` を呼ぶため、依存更新後はまず `dist/` を再ビルドしてから検証する。

```bash
bun run build && bun codesweep:fix
```

- `bun run build` — `dist/` を最新のソースから再生成する。`dist/` はリポジトリにコミットされる成果物（Git URL 配布のためにビルド済み JS が必要）なので、再ビルド後の差分は意図した変更
- `bun codesweep:fix` — フォーマット・lint を自動修正したうえで tsc / knip / syncpack / test を実行

エラーが発生した場合：

1. Context7 MCP で最新ドキュメントを確認し、修正を試みる
2. 修正が困難な場合は、該当パッケージのバージョンを `package.json` で戻し、`rm -rf node_modules bun.lock && bun i` で再インストールする
3. 再度 `bun run build && bun codesweep:fix` を実行し、すべてパスすることを確認する

## 最終レポート

全フェーズ完了後、Phase 2〜4 で生成した個別レポート（📦 パッケージ更新レポート / 🚀 ベストプラクティス移行 / 🧹 設定ファイル スリム化レポート）と Phase 5 の検証結果を、1つの「リフレッシュレポート」としてまとめてユーザーに提示する。更新も削除も無かったフェーズは「変更なし」と明記する。
